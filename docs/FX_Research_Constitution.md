# FX自動取引研究 基本方針（Research Constitution）

生成日時: 2026-02-28 14:40 UTC

本ドキュメントは、本研究全体に適用される不変の基本方針である。
個別戦略（tokyo_fix系、London/NY系など）を問わず、全ての実験・実装・運用判断は本方針に従う。

------------------------------------------------------------------------

# 1. 基本思想（最上位原則）

## 1.1 実行現実性最優先

-   Bid/Ask完全分離を厳守する
-   mid価格ベースの評価を最終判断に用いない
-   スプレッド・スリッページを必ず考慮する
-   「統計エッジ」と「実行可能エッジ」は別物である

## 1.2 未来参照の完全排除

-   推論時に未来情報を使用しない
-   学習時に使用する場合は明示する
-   同一バー内の高値安値順序問題を意識する
-   forced_exitや窓定義の曖昧さを排除する

## 1.3 過剰最適化の回避

-   小規模sweepに留める
-   多重検定リスクを常に意識する
-   walk-forward または holdout を必須とする
-   パラメータは説明可能な範囲に限定する

## 1.4 複雑性より頑健性

-   不要な高次MLモデルは使用しない
-   ロジスティック回帰など解釈可能モデルを優先する
-   「当てる」より「構造を捉える」設計を優先する

## 1.5 リスク制御前提

-   kill-switch前提で設計する
-   右尾依存モデルは疑う
-   Entry precisionが期待値を支配することを前提とする

## 1.6. 時間順序・価格系列規律

### 1.6.1 未来参照の厳格禁止（強化版）

-   entry時刻以前の価格情報を、直接・間接を問わず使用してはならない。
-   entryバー内のLow/Highを、順序未確定のままSL/TP判定に使用してはならない。
-   同一バー内でSL/TPが競合する場合は以下のいずれかを満たすこと：
    -   Low_Time / High_Time により順序が判定できる
    -   判定不能な場合は不利側優先
    -   もしくは「次バーから有効」とする
-   entry_time == exit_time かつ SL は原則バグとみなす（sanity必須）

### 1.6.2 Bid / Ask 系列固定原則

-   entry価格：Ask
-   exit価格：Bid
-   SL判定：Ask
-   TP判定：Bid
-   forced_exit：仕様で明示する
-   実験出力に使用系列を必ず記録する（未記録は不合格）

### 1.6.3 監視窓の分離原則

-   SL監視開始は entry直後とする。
-   model_exit監視窓とは独立させる。
-   監視窓の混同（例：mon_start = max(entry, exit_start)）を禁止する。
-   SL監視とexit監視は別ロジックとして実装する。

# 3. リスク管理方針

## 3.1 risk_fraction

正式採用値:

    risk_fraction = 0.5%

理由:

-   最大DDとCAGRのバランスが最良
-   holdoutでCAGR約17%、maxDD約15%
-   kill-switchと整合
-   証拠金維持率150%前後を維持可能

------------------------------------------------------------------------

## 3.2 証拠金維持率方針

-   運用目安維持率: 約150%
-   100%割れは危険域
-   強制ロスカットは約70%水準（ブローカー仕様）
-   0.5%超のrisk_fractionは維持率的に不適切

維持率制約は純粋な期待値評価より優先される。

# 4. 段階的スケーリング方針

運用開始後は段階的にスケールする。

## フェーズ1（初期検証）

-   極小ロット運用
-   仕様整合確認
-   約定・窓・forced_exit確認

## フェーズ2（小規模検証）

-   小ロット運用
-   ログ整合再確認

## フェーズ3（段階増加）

-   定期的に段階昇格
-   合格条件は「損益」ではなく「仕様整合」

昇格条件:

-   仕様通りの挙動
-   ログ完全一致
-   強制決済不備なし
-   Bid/Ask整合問題なし

------------------------------------------------------------------------

# 5. 最終原則

-   利益より生存
-   CAGRより破綻回避
-   統計より実行
-   複雑性より再現性

本研究は「破綻しない構造」を最優先とする。

# 6. 共通参照仕様（実装標準）

本章は全シリーズ共通の実装参照仕様である。


## 6.1 1分足仕様

## 入力データ
1分足（pkl.gz）を年ごとに読み込む（2019〜2025使用）:

```
/work/dataset/2019_1min.pkl.gz
/work/dataset/2020_1min.pkl.gz
/work/dataset/2021_1min.pkl.gz
/work/dataset/2022_1min.pkl.gz
/work/dataset/2023_1min.pkl.gz
/work/dataset/2024_1min.pkl.gz
/work/dataset/2025_1min.pkl.gz
```

DataFrame columns（例）:
- `Bid_Open, Bid_High, Bid_High_Time, Bid_Low, Bid_Low_Time, Bid_Close`
- `Ask_Open, Ask_High, Ask_High_Time, Ask_Low, Ask_Low_Time, Ask_Close`
- `Time_JST`（JST列）
- `Tokyo_Time, Index_Time` など

### 時刻の注意（重要）
- DataFrame の **indexはブローカー時刻（UTC+3）**
- 時刻の基準は **`Time_JST` を優先**して使うこと（JSTで日付・時刻窓を切る）
- もし `Time_JST` が無い／不正な場合は、index(UTC+3) から JST(+9) に **+6時間**して作る（repo慣習）


## 6.2 tick仕様

Parquet dataset:
- `ticks/USDJPY/`
- `year=YYYY/part-xxxxx.parquet`

必須スキーマ:
- `epoch_us: int64`  # UTC
- `bid: float`
- `ask: float`

読み込み要件:
- `epoch_us` 昇順で**完全ソート**（年/月跨ぎでも単調増加を保証）
- 対象期間は 2019-01〜2026-01 を全て扱える
- CLIで date range（JST日付）フィルタ指定可能
  - 例: `--date_from 2025-02-01 --date_to 2026-01-31`


## 6.3 検定仕様

probe1は以下のように実施します。

- コードはあなたが作成して、ダウンロード形式で出力する
- コードは`research/probe1/p001.py`みたいに置く
- 詳細ファイルは`research/probe1/outputs/P001/*.csv`に保存する。これはレポジトリに保存しない
- あなたにフィードバックする情報は標準出力に出す。`python research/probe1/p001.py | tee research/probe1/results/p001.log`みたいに実行する。research/probe1/resultsはレポジトリへ保存する
- 1分足データは上記の6.1 1分足仕様を参照のこと
- idxminでnaエラーが発生するので注意すること


## 6.4 実験仕様

実験は以下のように実施します。

- 仕様書は日本語/Markdownで作成し、ダウンロード形式で出力する。それをCodexが実装する
- 実験番号はE001みたいな形式とする
- 研究憲法を転記して、厳守するように書く。Codexは研究憲法を知らないので、転記は必要である
- 1分足仕様は6.1を転記、ticks仕様は6.2を転記すること
  - 1分足データについてはidxminでnaエラーが発生するので注意すること
- 以下はCodex宛の指示するファイル配置です。参考に共有します
  - Dockerfileは docker/Dockerfile を使用してください。
  - 実行ファイルは research/<実験シリーズ名>/e001.py とします。
  - research/<実験シリーズ名>にSpec.mdと使い方のREADME.mdは作成しましょう
  - 実行は以下のような形がいいです
```
docker run --rm \
  -v "$PWD:/work" \
  <実験シリーズ名> \
  python research/<実験シリーズ名>/e001.py \
  ...
```
  - tqdmによる進捗表示をお願いします
